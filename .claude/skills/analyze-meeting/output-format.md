# output-format — 리포트 파일 포맷과 마스킹

`analyze-meeting` skill이 최종 리포트를 작성할 때의 파일 포맷, 상단 메타데이터, 섹션 구조, 그리고 개인정보 마스킹 규칙.

## 저장 경로

```
projects/outputs/<project>/{YYYY-MM-DD-HHmm}-{mode}-report.md
```

- `<project>`: 입력 디렉터리 이름과 동일.
- `{YYYY-MM-DD-HHmm}`: 현재 로컬 시간, 분 단위.
- `{mode}`: `full` / `planning` / `tech` / `portfolio`.

## 리포트 전체 구조

```markdown
# {프로젝트명} — 종합 리포트 ({YYYY-MM-DD HH:mm})

- 프로젝트: {project}
- 입력 파일: projects/inputs/{project}/{...}
- 분석 모드: {mode}
- 참여 에이전트: {목록}
- 최우선 목표: 2026년 백엔드/프론트 신입 개발자 서비스/대기업 취업 성공

---

## 한 줄 요약
(취업 임팩트 관점에서 현 기획의 현주소 한 문장)

## 정(正) — 각 에이전트 원문 (축약 없음)

### tech-lead-mentor
(원문 전체 그대로, 마스킹만 적용)

### target-user-persona
(원문 전체 그대로, 마스킹만 적용)

### swm-reviewer
(원문 전체 그대로, 마스킹만 적용)

### peer-competitor
(원문 전체 그대로, 마스킹만 적용)

## 반(反) — 충돌 지점
(synthesis-framework의 반(反) 규약에 따라)

## 합(合) — 취업 임팩트 기준 종합 권고
(synthesis-framework의 합(合) 규약에 따라)

## 팀원별 취업 레버리지 점검
(synthesis-framework의 점검표 규약에 따라)

## 다음 회의 전까지 팀이 답해야 할 질문
1. ...
2. ...
3. ...
```

모드가 `full`이 아닐 때는 참여하지 않은 에이전트의 `### {agent-name}` 섹션을 생략한다(빈 섹션을 남기지 않는다).

## 한 줄 요약 작성 규칙

- **취업 임팩트 관점**에서 현 기획의 현주소를 한 문장으로.
- 격려·완곡어 금지. 예시:
  - "현 기획은 취업 임팩트 관점에서 약함. 면접 카드 2장이 모호하고 팀원 2명의 지분이 불분명."
  - "백엔드 루트는 강함(면접 카드 2장 확보), 프론트 루트는 약함(포지셔닝 미결)."

## 개인정보 마스킹 규칙

리포트 저장 단계에서만 적용한다. 서브에이전트 호출 시(execution.md 단계 3)에는 원문 그대로 전달한다.

### 적용 대상과 형태

| 대상 | 원본 예시 | 마스킹 결과 |
| --- | --- | --- |
| 한국식 이름 (2~4자) | 김철수, 박지연 | `김OO`, `박OO` |
| 전화번호 | 010-1234-5678 | `010-XXXX-XXXX` |
| 이메일 | abc@example.com | `xxx@xxx.com` |
| 주민등록번호 | 900101-1234567 | `XXXXXX-XXXXXXX` |

### 정(正) 섹션에 원문을 붙일 때

- 마스킹 외의 편집은 금지. 서브에이전트가 쓴 문장을 요약·재구성하지 않는다.
- 서브에이전트가 이미 마스킹된 형태로 이름을 언급했다면 그대로 둔다.
- 코드블록·표·인용 블록의 구조를 변경하지 않는다.

### 예외

- 공인의 실명(예: 기존 서비스의 대표 이름이 벤치마크로 언급된 경우)은 마스킹하지 않는다. 판단이 애매하면 마스킹 우선.
- RAW DATA에 팀원 이름 대신 역할명(예: "PM", "프론트엔드")만 있으면 마스킹할 대상이 없으므로 그대로 둔다.

## 저장 후 사용자 출력

리포트 저장이 끝나면 현재 대화 응답에서 다음 두 가지만 짧게 전달한다.

1. 저장 경로(클릭·복사 가능한 형태).
2. 리포트의 "한 줄 요약" 본문.

전체 리포트를 대화창에 그대로 복사하지 않는다(사용자가 파일을 직접 열어 읽는다).
