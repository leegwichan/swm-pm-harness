# execution — 실행 절차 6단계

`analyze-meeting` skill이 호출되면 메인 Claude는 아래 단계를 **순서대로** 수행한다. 인자 파싱과 검증은 [`invocation.md`](./invocation.md) 참조.

## 1) 입력 로드

1. [`invocation.md`](./invocation.md)의 검증 규약을 먼저 통과시킨다.
2. `input-path`가 주어지면 그 단일 파일을 `Read`해 RAW DATA로 사용한다.
3. 생략되었다면 **누적 자료 전체 분석**을 기본값으로 한다:
   - `Glob`으로 `projects/inputs/<project>/**/*.{md,html,txt}`(존재하는 확장자만)를 조회한다.
   - 결과를 **파일명 오름차순(파일명에 날짜가 인코딩되어 있다는 가정)** 으로 정렬한다. 동일 날짜 그룹 안에서는 의미상 본문 → 요약 → 피드백 순으로 자연스럽게 읽히도록 파일명 정렬을 그대로 따른다. 결정 불가하면 수정시간 오름차순을 보조 기준으로 사용한다.
   - 각 파일을 순서대로 `Read`한 뒤 **하나의 RAW DATA 블록으로 이어붙인다**. 각 파일 사이에는 다음 형식의 명확한 구분자를 둔다:
     ```
     === FILE: <상대경로> ===
     (파일 내용)
     === END FILE: <상대경로> ===
     ```
   - 4) 종합 단계의 합본 RAW DATA 블록은 서브에이전트 호출 시 그대로 동봉한다(아래 단계 3 참조).
4. RAW DATA 합본이 너무 커서 단일 호출에 부담이라고 판단되면 **임의 절단·요약하지 말고**, 사용자에게 "특정 기간 또는 파일 범위로 좁힐지" 되묻는다. 자동 축약 금지.

개인정보(이름/연락처/이메일 등)가 포함되어 있어도 **서브에이전트에게는 원문을 그대로 전달**한다. 마스킹은 최종 리포트 저장 단계에서만 적용([`output-format.md`](./output-format.md) 참조).

## 2) 모드에 따라 서브에이전트 선택

모드 표는 [`invocation.md`](./invocation.md)에 정의되어 있다. 그 표의 "호출할 에이전트" 컬럼을 그대로 따른다.

## 3) 서브에이전트 병렬 호출 (중요)

- **반드시 단일 메시지에서 여러 Agent tool call을 동시에 발행**하여 병렬 실행한다. 순차 호출 금지. 목적: 에이전트 간 의견 오염 방지.
- 각 서브에이전트에게는 단계 1에서 만든 **합본 RAW DATA(여러 파일이 `=== FILE: ... ===` 구분자로 이어진 형태)** 와, "당신의 페르소나 정의에 따라 단독으로 분석하라. 다른 에이전트의 의견은 주어지지 않는다"는 지시를 함께 전달한다. 여러 파일이 시간순으로 누적된 자료라는 사실을 명시해, 에이전트가 시간 흐름·기획 변경·반복 지적 사항을 추적할 수 있게 한다.
- 각 에이전트의 출력 포맷(자기 정의 파일에 명시됨)을 준수하라고 명시한다.
- **각 서브에이전트의 응답 전체**를 나중에 리포트 정(正) 섹션에 원문 그대로 붙일 것이므로, 누락 없이 수신한다.

### 서브에이전트 호출 프롬프트 템플릿

각 에이전트에게 동일하게 전달하는 프롬프트:

> 당신은 `{agent-name}` 페르소나다. 자신의 정의 파일(`.claude/agents/{agent-name}.md`)에 있는 페르소나 배경, 성격, 판정축, 핵심 질문, 출력 포맷을 **엄격히 준수**하라.
>
> 아래 RAW DATA는 `projects/inputs/<project>/`에 누적된 **여러 회의·기획·피드백 파일을 시간순으로 이어붙인 합본**이다. 각 파일은 `=== FILE: <path> ===` 구분자로 시작하고 `=== END FILE: <path> ===`로 끝난다. 시간 흐름에 따른 **기획 변화·반복되는 지적·해결되지 않은 쟁점**을 추적하면서 분석하라.
>
> 당신의 출력 포맷에 맞춰 리포트를 **완전한 형태로** 작성하라. 다른 에이전트의 의견은 주어지지 않으며, 당신 단독 판단만 요구된다.
>
> **팀의 최우선 목표**는 "SWM에서의 활동을 바탕으로 2026년 백엔드/프론트 신입 개발자 서비스/대기업 취업에 성공한다"임을 인지하고, 당신의 페르소나 관점에서 이 목표에 기여 또는 저해하는 요소를 판단하라. 단, 당신의 페르소나 톤에서 벗어나지 말 것(예: `target-user-persona`는 취업을 알지 못한다 등 페르소나 제약은 유지).
>
> 격려·완곡어·추측을 금지한다. 모르는 기술 사실은 "확인 필요"로 표기한다.
>
> --- RAW DATA 시작 ---
> {합본 input 전문 — 모든 파일이 시간순으로 이어진 형태}
> --- RAW DATA 끝 ---

## 4) 종합 (synthesis-framework 로드)

- `.claude/skills/synthesis-framework/SKILL.md`를 `Read`로 불러 그 규약을 따른다.
- 합(合) 단계는 반드시 "2026년 백엔드/프론트 신입 취업 성공" 기준으로 판정한다.
- **정(正) 섹션에는 각 에이전트의 응답 원문을 축약 없이 전문으로 수록**한다. 요약본을 정에 넣지 않는다.

## 5) 리포트 저장 (다중 파일)

한 번의 실행 결과는 하나의 디렉터리로 저장된다. 에이전트별 원문 파일과 종합 파일이 **모두 별도 파일**로 생성된다.

- 저장 디렉터리: `projects/outputs/<project>/{YYYY-MM-DD-HHmm}-{mode}/`
  - `projects/outputs/<project>/`가 없으면 먼저 생성(자동 생성 허용 — 입력 디렉터리가 이미 존재하므로).
  - 해당 타임스탬프 디렉터리도 `Write`로 파일을 만들면서 자연 생성되지만, 명시적으로 `mkdir`해도 된다.
  - 타임스탬프는 현재 로컬 시간, 분 단위까지.
  - 동일 분 단위 디렉터리가 이미 존재하면 뒤에 `-2`, `-3` 접미사.
- 저장 파일:
  1. 참여한 각 에이전트마다 한 파일씩: `{agent-name}.md` (예: `tech-lead-mentor.md`).
     - 내용은 에이전트 응답 원문을 **축약 없이** 그대로. 마스킹만 적용.
     - 참여하지 않은 에이전트의 파일은 **만들지 않는다**.
  2. 종합 파일: `synthesis.md`.
     - 한 줄 요약 + 정(正) 섹션(에이전트 파일 링크) + 반(反) + 합(合) + 팀원별 점검 + 다음 회의 질문.
     - 에이전트 원문을 `synthesis.md`에 **중복 복사하지 않는다**(링크로만 참조).
- 저장 순서: 에이전트 파일을 먼저 만들고 마지막에 `synthesis.md`를 만든다. `Write`를 각 파일마다 호출.
- 저장 후 사용자 출력: [`output-format.md`](./output-format.md)의 "저장 후 사용자 출력" 규약에 따라 디렉터리 경로 + 파일 목록 + 한 줄 요약만 전달.

리포트 내부 포맷은 [`output-format.md`](./output-format.md) 참조.

## 6) 개인정보 마스킹

마스킹 규칙 상세는 [`output-format.md`](./output-format.md)에 정의되어 있다. 서브에이전트 호출 시(단계 3)에는 마스킹하지 않으며, 리포트 저장 시(단계 5)에만 적용한다. **에이전트 개별 파일과 `synthesis.md` 모두**에 동일하게 적용한다.
