# synthesis-framework

`analyze-meeting` 오케스트레이터가 수집한 서브에이전트 출력들을 종합할 때 따르는 규약이다.

## 최우선 목표 (모든 종합의 판정 기준)

> **"SWM에서의 활동을 바탕으로 2026년 백엔드/프론트 신입 개발자 서비스/대기업 취업에 성공한다."**

이 목표는 종합 리포트의 **합(合)** 단계에서 유일한 판정 기준이다. "기술적으로 재미있다", "유저가 좋아한다", "평가위원이 점수를 준다"는 **보조 지표**이며, 이 보조 지표들이 취업 임팩트로 환전되는지를 확인하는 것이 종합의 역할이다.

## 종합 원칙

1. **취업 임팩트가 판정축이다.** 다른 기준들(기술 재미, 평가 점수, 유저 PMF)은 취업 임팩트로 환전되는 한도 내에서 가치를 갖는다.
2. **충돌은 숨기지 않는다.** 에이전트 간 의견 충돌은 반(反) 섹션에서 명시적으로 드러낸다.
3. **톤은 보존한다.** swm-reviewer의 드라이함, peer-competitor의 시큰둥함, target-user-persona의 관심 없음을 종합 단계에서 세탁하지 않는다.
4. **합의 원칙은 집중이다.** 전 영역에서 동시에 도전하지 않는다. "백엔드 루트 / 프론트 루트에서 각자 면접용 깊이 1개씩"이 기본.

## 3단 구조 + 팀원별 점검

### 정(正) — 각 에이전트의 원문 (축약 없음)

- **서브에이전트 응답 원문을 그대로 붙인다.** 요약본을 넣지 않는다.
- 마스킹 외의 편집은 금지.
- 각 에이전트별 `### {agent-name}` 헤더 아래 원문 전체.

### 반(反) — 에이전트 간 충돌 지점

에이전트 간 의견이 **실제로 부딪히는 지점**만 나열한다. 억지로 만들지 않는다. 각 충돌은 **A ↔ B** 형태로 기술한다.

아래 3개 프리셋 충돌축을 **반드시 체크**한다(해당 모드에서 두 에이전트가 모두 호출된 경우에 한함).

#### 프리셋 충돌축 1 — tech-lead-mentor ↔ peer-competitor

**"안전한 MVP vs 차별화 도전"**

- tech-lead-mentor는 3개월(UT·발표 준비·발표 포함) 안에 실사용 가능한 완성도로 나올 스코프를 원한다.
- peer-competitor는 GitHub 수천 개와 구분될 차별화를 원한다.
- 안전하면 지루하고, 차별화하면 터진다.
- **취업 관점 중재**: 차별화 축 **1개**만 살리면 충돌이 해소된다. 그 1개가 면접 30분 서사가 된다.

#### 프리셋 충돌축 2 — target-user-persona ↔ swm-reviewer

**"유저가 원하는 단순함 vs 심사위원이 원하는 기술 난이도"**

- target-user-persona는 3초 안에 이해되는 단순함을 원한다.
- swm-reviewer는 세금 지원의 명분이 되는 기술 난이도를 원한다.
- UI/UX와 백엔드 난이도의 긴장으로 나타난다.
- **취업 관점 중재**: 레이어 분리. 유저에게는 단순, 평가·면접용으로는 내부 깊이. 백엔드 지원자는 "내부 깊이"를, 프론트 지원자는 "체감 완성도·성능"을 각자 자기 면접 카드로 가져간다.

#### 프리셋 충돌축 3 — tech-lead-mentor ↔ swm-reviewer

**"현실적 범위 vs 도전적 범위"**

- tech-lead-mentor는 "돌아가는 것"을 우선한다.
- swm-reviewer는 "3개월 압축 일정의 세금 지원 결과물"로 타당한 난이도를 요구한다.
- 이 긴장이 **적정 난이도 밴드**를 만든다.
- **취업 관점 중재**: "돌아가면서 면접에서 말할 거리가 있는" 지점이 답이다. 평가 점수와 취업 점수는 완전히 동일하지 않다는 걸 기록해둔다.

#### 그 외 충돌

프리셋 외에도 실제 충돌이 있으면 자유롭게 추가한다. 호출된 에이전트가 적어 체크할 수 없으면 "이번 모드에서는 체크 불가"라고 표기하고 건너뛴다.

#### 충돌이 전혀 없을 때

> 이번 분석에서는 주요 충돌이 없었음. 에이전트들이 대체로 같은 방향을 지적함: (공통 지적사항 요약)

단, 충돌이 없다는 사실 자체가 "기획이 너무 밋밋해서 아무도 이의제기할 구석이 없다 → 취업에도 쓸 게 없다"는 신호일 수 있음을 덧붙인다.

### 합(合) — 취업 임팩트 기준 종합 권고

합(合) 섹션은 **취업 임팩트**를 축으로 삼는다. 다른 기준과 충돌할 때 취업 임팩트가 우선이다.

#### 합의 4원칙

##### 원칙 1 — 면접 카드 2~3개에 집중

- 팀 프로젝트 하나로 **팀원 각자가 자기 면접에서 30분 떠들 카드**가 2~3개는 나와야 한다.
- 카드 1장 = "왜 A 대신 B인가"로 정리되는 의사결정.
- 전 영역에서 카드를 만들지 말고, **깊이 있는 카드 2~3개**만 만든다.

##### 원칙 2 — 백엔드 루트 / 프론트 루트의 카드를 분리해서 본다

- 팀 안에 백엔드 지원자와 프론트 지원자가 섞여 있다.
- **같은 프로젝트에서 백엔드 면접용 카드와 프론트 면접용 카드가 각각 나와야 한다.**
- 백엔드 카드 예시: 성능 최적화, 분산 처리, 데이터 파이프라인, 인프라 자동화 (실제 기획에 따라 조정)
- 프론트 카드 예시: 렌더링 성능, 접근성, 상태 관리, 애니메이션·인터랙션, 실시간 UI (실제 기획에 따라 조정)

##### 원칙 3 — "차별화"는 2~3개 결정에 집중, MVP 나머지는 안전 스택

- peer-competitor가 요구하는 차별화는 **프로젝트 전체 이상함**이 아니다. **의사결정 2~3개**면 충분하다.
- 나머지 90%는 검증된 라이브러리·패턴·관리형 서비스로 빠르게 채운다.
- 수료·취업이 끝나기 전에 **돌아가는 것**이 가장 큰 차별화다.

##### 원칙 4 — 2026 신입 채용 시장의 관심 주제와 연결

- 면접관이 관심 가질 주제와 이 프로젝트의 의사결정이 **연결**되는지 확인한다.
- 연결되지 않으면 의사결정 포인트 자체를 이동시키거나, 같은 구현 안에서 **다른 이야기 각도**를 찾는다.
- (2026 시장의 구체 트렌드 언급은 추측이 될 수 있으므로 "확인 필요" 표기 원칙을 지킨다. 단, "지원할 회사의 JD와 기술 블로그를 팀이 직접 읽고 맞추라"는 지시는 기본값으로 넣는다.)

#### 합 섹션 구성

```markdown
## 합(合) — 취업 임팩트 기준 종합 권고

### 한 줄 판정
(현 기획은 취업 임팩트 관점에서 [강함 / 보통 / 약함]. 이유 1~2문장.)

### 스코프 결정

- 끝까지 살릴 것 (면접 카드로 환전 가능한 핵심):
- 유보 (추후 결정):

각 카드에 대해: 이 카드로 어떤 30분 서사를 만드는가 / 지원할 회사 유형과 어떻게 연결되는가.

### 차별화 레버
```

### 팀원별 취업 레버리지 점검

4명 팀이라면 각자 **자기 지분으로 자기 언어로 면접에서 말할 수 있는가**를 체크한다.

```markdown
## 팀원별 취업 레버리지 점검

| 팀원 역할 | 지원 루트(백/프) | 이 프로젝트에서의 지분 | 면접 카드 (본인이 쓸 수 있는 것) | 부족한 것 |
| --- | --- | --- | --- | --- |
| ... | 백엔드 | ... | ... | ... |
| ... | 프론트 | ... | ... | ... |
| ... | ... | ... | ... | ... |
| ... | ... | ... | ... | ... |
```

입력에서 팀원 정보를 추출할 수 없으면 "입력에 팀원 역할이 명시되어 있지 않음. 다음 회의에서 정의 필요"라고 기록한다. 임의로 이름·역할을 만들지 않는다.

### 다음 회의 전까지 팀이 답해야 할 질문

- 번호가 붙은 구체적 질문 3~5개.
- 각 질문은 **취업 임팩트**와 연결되어 있어야 한다.
- "검토해보자" 같은 모호한 표현 금지.

## 종합 품질 체크리스트

리포트를 마치기 전 메인 Claude는 다음을 스스로 점검한다.

- [ ] 정(正) 섹션에 각 에이전트의 응답이 **원문 전체로** 들어갔는가? (요약 금지)
- [ ] 정(正) 섹션에서 각 에이전트의 **톤**이 보존되었는가?
- [ ] 반(反) 섹션에서 프리셋 충돌축 3개를 **모두 체크**했는가? (해당 모드 기준)
- [ ] 반(反) 섹션에서 **없는 충돌을 억지로 만들지 않았는가**?
- [ ] 합(合) 섹션에 **"면접 카드 2~3장"**이 구체적으로 명시되었는가? 백엔드/프론트 루트 구분이 있는가?
- [ ] 합(合)의 모든 권고가 **취업 임팩트 기준**으로 설명되는가? (평가 점수만 보고 판단한 부분은 없는가)
- [ ] 팀원별 취업 레버리지 점검 표가 있는가? (없으면 "입력에 팀원 역할 없음"을 명시했는가)
- [ ] 합(合) 섹션에 **구체적 액션**(자를 것 / 살릴 것)이 있는가? "검토해보세요" 금지.
- [ ] 개인정보가 마스킹되었는가?
- [ ] 라이브러리·API·벤치마크·2026 시장 트렌드 등 추측성 사실에 **"확인 필요"** 표기가 되어 있는가?
