# SWM Reviewer — 평가

## 한 줄 평가

현 시점 제출물 기준 **중간평가는 어렵게라도 통과 가능**, **최종 수료평가는 현 스코프대로면 불통 리스크 매우 높음**. 사유 4개. 가장 큰 문제는 "데모 당일 라이브 동작 = STT 자막"에 그치는데, 4-30 회의에서 분석 항목을 18개 이상으로 추가 확장해 스코프가 폭발한 것.

## 기술적 난이도 타당성

- **현재 난이도 수준: [과욕]** — 4-23 시점은 [적정] 수준이었으나, 4-30 회의 이후 [과욕]으로 급격히 이동.
- **근거:**

  - 4-30 회의에서 "분석 탭"에 다음 항목들이 동시에 ⭐⭐ 이상 우선순위로 박힘:

    - LLM judge 쟁점별 우세 판정

    - 반박 직접성 (임베딩 유사도 + LLM 판단)
    - 발화 점유율, WPM, TTR, 필러 빈도, 말 끊기 탐지(타임스탬프 겹침), 응답 지연
  - **각 항목이 독립적으로 1~2주 R&D가 필요한 LLM 평가 문제다.** 정답셋 없이 LLM-as-judge로 메트릭 18개를 동시에 신뢰 가능한 수준으로 만들 수 있다고 보지 않음. 이OO 멘토가 "LLM judge로 정답셋 비교가 현실적"이라고 답한 건 **하나를** 검증하는 방법이지 18개를 한꺼번에 검증하라는 뜻이 아님.

  - "실시간 승리 확률 like 체스"는 **검증 가능한 ground truth가 존재하지 않는다.** 체스는 엔진 평가 함수가 합의된 정답이지만, 토론 승률 예측은 학술적으로도 미해결. 이를 분석 탭의 ⭐⭐⭐ 핵심 지표로 박은 것은 평가 당일 가장 먼저 무너질 항목.

- **조정 필요 부분:**
  1. **분석 항목 18개 → 3개로 컷.** 추천 우선순위: ① 주장-근거-반박 트리 추출 ② 발화 점유율/통계성 ③ LLM judge 쟁점별 우세 판정 (사후, 단일 점수가 아니라 텍스트 총평).
  2. "실시간 승리 확률"은 **삭제** 또는 "데모 전용 시각화"로 격하. 평가위원 100% 이걸 짚어 묻는다. 답변 못하면 끝.
  3. "논리적 오류 탐지 4종"은 **삭제** 또는 **단일 종(예: 허수아비)**로 컷. 4종을 한국어 토론 컨텍스트에서 신뢰성 있게 분류하는 평가셋이 없다.
  4. 음성 입력 기반 토론 분석은 **세다(아카데미식) 턴 기반에 완전 고정**. 자유토론 화자 분리는 기획서대로 2차로 완전 보류.

## 불통과 리스크 (가장 중요)

현 시점 기준 평가에서 떨어질 가능성이 있는 사유를 우선순위대로:

1. **(CRITICAL) 분석 탭 항목 폭발 — 18개 메트릭 중 절반 이상이 데모 당일 모의 데이터로 보일 것.** 평가위원은 분석 탭 클릭하자마자 "이 점수는 어떻게 계산되나요?", "정답셋이 있나요?", "샘플 데이터인가요 실 토론인가요?"를 묻는다. 답변 막히는 순간 끝. **4-30 회의에서 ⭐⭐⭐ 항목들 거의 전부가 ground truth가 없는 LLM 평가 영역이다.**

2. **(CRITICAL) 마일스톤 캘린더 부재.** 4-23 기획서에 "1순위로 다음 회의에 정한다"고 박힌 항목이 보름이 지난 4-30까지 회의록에 등장하지 않음. 중간평가에서 제출하라고 요구하는 일정 산출물 자체가 없을 가능성. **즉 중간평가 통과조차 의심스러움.**

3. **(HIGH) 차별점 한 줄이 검증 불가능한 주장으로만 구성됨.** "토론 복기에 도움", "수작업의 자동화" — 2026년 시점에 회의록 자동화 도구는 범용 도구로 다수 존재(확인 필요). "왜 토론 도메인 특화가 필요한가"의 답이 한 줄로 안 나옴. 평가위원 첫 질문에서 막힘.

4. **(HIGH) Bus Factor 1.** 도메인 지식, 인터뷰 라인, DT 유저풀 모두 콜리 1인 집중. 4-23 기획서가 명시. 4-27/4-30 회의록에서 쿼찬 등장 없음. **확인 필요: 쿼찬은 합류했는가? 합류 안 했다면 평가에서 "팀 구성이 3인이냐 2인이냐"부터 문제 제기됨.**

5. **(HIGH) 데모 환경·실패 시나리오 정의 없음.** 4명이 라이브로 토론하는 것을 시연 시간 내에 보여주는 것은 사실상 불가능(시연 5분 내 토론이 일어날 수 없음). 사전 녹음 재생으로 갈 텐데 그러면 "실시간성"이 가짜가 됨. 평가위원이 마이크 입력 ON 상태에서 직접 1분 발화해보겠다고 하면 그게 라이브 검증이고 STT 정확도가 노출됨.

6. **(MEDIUM) 산출물(세특/카드뉴스/판정 보고서) 3종 모두 LLM 출력물.** 평가위원은 "이게 GPT 래퍼와 어떻게 다른가"를 반드시 묻는다. 4-23 기획서 O-205에서 본인들이 인식한 "래퍼 비판" 리스크가 그대로 남아 있다. "유닛 분할 → 의도 분류 → Rebuts 매칭 → 세특 생성+인용"이라는 4단계 파이프라인 설계는 인식만 됐고 구현 우선순위는 분석 탭 18개에 밀림.

7. **(MEDIUM) 영업·법규 리스크.** 세특 AI 생성 — 4-23 기획서 O-103에서 본인들이 명시한 교육부 가이드라인 위반 가능성. 4-27 회의에서 "가이드라인 수준으로" 한 줄 처리. **확인 필요지만, 평가위원이 "교육청 컴플라이언스 검토했나요"를 물으면 답이 없음.** 카드뉴스 자동 생성 시 동아리 로고·이미지 저작권 문제(인스타 nogotte/gorandoran 레퍼런스 활용 시).

8. **(MEDIUM) 4-23 기획서 R-04 "전부 만들지 않는다"가 4-30에 깨짐.** 본인들이 4-23에 "원칙"으로 천명한 것을 일주일 만에 어김. 평가위원이 회의록 비교하면 바로 보임.

## 통과를 위해 다음 평가 전까지 해야 할 것

(우선순위대로. 격려 없이 지시형으로.)

1. **분석 탭 18개 → 3개로 즉시 컷.** 컷 기준은 (a) 한국어 토론 5~10건으로 정답셋 만들 수 있는가 (b) 데모 당일 라이브로 동작 가능한가 (c) 평가위원 검증 질문에 30초 안에 답할 수 있는가. **기한: 다음 회의 (5월 첫 주). 결정 책임자: 콜리.**

2. **마일스톤 캘린더 작성.** SWM M1 / M2 / 중간평가 / M3 / 수료 데모데이 일자를 캘린더에 박고, 각 마일스톤마다 검증 가능한 산출물 1개씩 정의. "구현 완료"가 아니라 "사용자 A가 시나리오 B를 수행해 결과 C를 본다" 형식으로. **기한: 5월 첫 주. 책임자: 팀 전체.**

3. **쿼찬 합류 확인 + BE 책임자 1인 지정 문서화.** 4-23 기획서 O-201이 보름째 미해결. 합류 안 했으면 2인 팀으로 SWM 운영부에 변경 신고했는지 확인. **기한: 즉시. 책임자: 콜리.**

4. **데모 환경·시나리오 결정.** 사전 녹음 5건 + 라이브 1건(평가위원 직접 발화 1분) 같은 구체 시나리오로 결정. STT API 1순위·2순위 + Fallback 시점 정의. **기한: M1 종료 전. 책임자: BE.**

5. **차별점 한 줄을 검증 가능한 형태로 재작성.** 위 "재작성 제안"의 (A)+(B)+(C) 중 2개 이상이 데모에서 동작하도록 스코프 고정. 광고 카피 ("토론 복기에 도움") 금지. 숫자 또는 동작 명시. **기한: 5월 첫 주. 책임자: 콜리.**

6. **STT 정확도 사전 검증.** 이OO 멘토 권고대로 A4 반장 텍스트 5개 사전 녹음 → STT 결과 비교 → 50% 미만이면 STT 엔진 또는 녹음 환경 변경. 1순위 한국어 토론(다자 화자) 데이터에서 검증. **기한: M1 첫 주. 책임자: BE+AI.**

7. **카드뉴스/세특을 핵심 데모에서 빼는 방안 검토.** 카드뉴스는 4-29 FE 피드백에서도 "정말로 필요한지?"라고 본인들이 의문 제기. 세특은 법규 리스크. 둘 다 핵심 면접 카드와 무관 — Live + Report (분석 3개) + 단일 산출물 1개로 컷. **기한: 5월 첫 주.**

8. **취업 카드 ↔ 평가 통과 충돌 지점 명시.** "FE가 분석 탭 차트(DOM/SVG/Canvas) 비교 경험을 면접 카드로 가져가고 싶다"는 동기와, "데모 당일 안정 동작" 사이에 충돌 발생 가능. FE는 분석 탭 차트 1종만 깊게 파고 나머지는 단순 차트 컴포넌트 사용으로 결정. 면접 카드는 깊이 있는 1개로 충분. **결정 책임자: FE 본인.**

## 평가위원 추가 메모 (페르소나 외 단일 코멘트)

시안 디자인 수준은 동일 페르소나로 본 다른 팀 평균 이상이다. 4-30 dashboard/analytics/individual의 시각적 완성도는 높다. 그러나 **평가위원은 디자인을 평가하지 않는다.** 디자인이 좋을수록 "이게 영상 합성된 시안이냐 라이브 동작이냐"를 구분해서 묻는다. 시안에서 보이는 메트릭을 라이브로 백킹하는 데이터 파이프라인이 4-30 시점에 0% 구현 상태인 것이 가장 큰 문제다. 보름간 누적된 회의록의 결정 사항 대비, 실제 코드·데이터·평가셋 산출물이 한 건도 없는 점이 단계평가에서 직접 노출된다.
