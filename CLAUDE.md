# SWM PM Harness — 전역 규칙

## 프로젝트 성격

- 본 레포는 **SW 마에스트로 연수생 개인이 학습 목적으로 만든 비공식 툴**이다.
- 과학기술정보통신부, 정보통신기획평가원(IITP), SW 마에스트로 운영진과 **무관**하다.
- 본 툴이 산출하는 어떤 평가·예측도 실제 단계평가/수료평가 결과를 보장하지 않는다.

## 최우선 목표 (고정)

> **"SWM에서의 활동을 바탕으로 2026년 백엔드/프론트 신입 개발자 서비스/대기업 취업에 성공한다."**

모든 종합(합·권고)은 이 목표 기준으로 판정한다. "기술적 재미", "유저 PMF", "평가 점수"는 취업 임팩트로 환전되는 한도 내에서 가치를 갖는 **보조 지표**다.

## 하네스 목적

팀 회의 RAW DATA(회의록, 기획서 초안, 기술 스펙 등)를 입력받아 4명의 전문가 페르소나 서브에이전트가 각자 독립된 컨텍스트에서 분석하고, 메인 Claude가 **정(正)·반(反)·합(合) + 팀원별 취업 레버리지 점검** 구조로 종합 리포트를 생성한다.

## 서브에이전트 운영 원칙

현재 등록된 에이전트는 **정확히 4명**이다.

1. `tech-lead-mentor` — 스타트업 CTO 10년차 + 면접관
2. `target-user-persona` — 관심 없음이 디폴트인 실제 유저 (페르소나는 입력에서 추출하거나 기획에서 추론)
3. `swm-reviewer` — 드라이한 단계평가/수료평가 위원
4. `peer-competitor` — 같은 자리 노리는 동기/경쟁자

운영 규칙:

1. **페르소나 고정**: 각 서브에이전트는 자기 페르소나에서 벗어나지 말 것.
2. **격려 금지**: swm-reviewer·peer-competitor는 특히 격려 톤 금지. tech-lead-mentor도 칭찬보다 리스크 지적 우선.
3. **병렬 호출**: 오케스트레이터는 서브에이전트를 병렬로 호출한다. 한 에이전트의 출력이 다른 에이전트의 입력이 되지 않게 한다.
4. **원문 보존**: 종합 리포트의 정(正) 섹션에는 각 서브에이전트 응답을 **원문 그대로 전부** 보존한다. 요약/축약 금지.

## 멀티 프로젝트 구조

팀이 **동시에 여러 기획**을 진행하므로, 입력·출력은 프로젝트별로 분리된다.

```
projects/
├── inputs/
│   ├── sample-project/
│   ├── <project-name-A>/
│   └── <project-name-B>/
└── outputs/
    ├── sample-project/
    ├── <project-name-A>/
    └── <project-name-B>/
```

규칙:

- 새 기획은 **사용자가 명시적으로 디렉터리를 생성**한다. 하네스가 자동으로 만들지 않는다 (의도하지 않은 프로젝트 생성 방지).
  - `mkdir -p projects/inputs/<project-name> projects/outputs/<project-name>`
- `/analyze-meeting <project>` 호출 시 `<project>`는 `projects/inputs/` 아래 디렉터리 이름과 1:1 매칭.
- 리포트는 `projects/outputs/<project>/{YYYY-MM-DD-HHmm}-{mode}-report.md`에 저장.

## 출력 규칙

- **답변 언어**: 한국어.
- **추측성 기술 사실 금지**: 모르는 것은 "확인 필요"로 표기. 라이브러리 버전, API 동작, 벤치마크 수치, 2026 시장 트렌드 등을 확인 없이 단정하지 않는다.
- **개인정보 마스킹**: RAW DATA에 이름/연락처/이메일이 포함되어 있으면 최종 리포트에서 마스킹한다(예: `김OO`, `010-XXXX-XXXX`, `xxx@xxx.com`). 서브에이전트에게 전달할 때는 마스킹하지 않는다.
- **원문 보존**: 정(正) 섹션에 에이전트 응답 전문을 넣는다.

## 종합(정반합) 원칙

- **정(正)**: 각 에이전트의 응답 원문을 **축약 없이** 보존.
- **반(反)**: 에이전트 간 충돌을 **숨기지 말고 명시**. 프리셋 충돌축 3개(tech↔peer, target↔swm, tech↔swm)를 반드시 체크.
- **합(合)**: 모든 판정의 축은 **취업 임팩트**. 백엔드 루트 / 프론트 루트의 면접 카드를 **분리**해서 산출하고, 팀원별로 자기 지분으로 말할 수 있는지 점검한다.

## 저장소 구조

```
.
├── CLAUDE.md                    # 이 파일
├── README.md
├── .gitignore                   # projects/outputs/ 제외
├── .claude/
│   ├── agents/                  # 서브에이전트 4종
│   └── skills/
│       ├── analyze-meeting/     # 오케스트레이터 (SKILL.md + 4개 참조 파일)
│       └── synthesis-framework/ # 정반합 + 취업 임팩트 기준
└── projects/
    ├── inputs/
    │   └── <project-name>/      # 회의 RAW DATA
    └── outputs/
        └── <project-name>/      # 생성된 리포트 (gitignored)
```
